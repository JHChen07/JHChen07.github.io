产生跨域的原因

跨域问题是浏览器同源策略限制，当前域名的js只能读取同域下的窗口属性

同源策略

定义

如果两个页面拥有相同的协议（protocol），端口和主机，那么这两个页面就属于同一个源（origin）
一个网站的网址组成包括协议名，子域名，主域名，端口号。比如https://github.com/，其中https是协议名，www是子域名，github是主域名，端口号是80，当在页面中从URL请求数据时，如果这个URL的协议名、子域名、主域名、端口号任意有一个不同，就会产生跨域问题。即使是在 http://localhost:80/ 页面请求 http://127.0.0.1:80/ 也会有跨域问题。

同源策略用意

同源策略限制了从同一个源加载的文档或脚本如何与来自另一个源的资源进行交互。这是一个用于隔离潜在恶意文件的重要安全机制。

解决跨域问题

解决跨域问题有以下几种方式：

  1. 使用JSONP
  2. 服务端代理
  3. 服务端设置Request Header头中Access-Control-Allow-Origin未指定可获取数据的域名

    JSONP的解决方式

  4. 原理
    JSONP解决跨域问题的原理是，浏览器的script标签是不受同源策略限制（可以在网页中设置script的src属性为cdn服务器中静态文件的路径），这样就可以使用script标签从服务器获取数据，请求时添加一个参数为callback=？，？号是要执行的回调方法。

  5. 前端实现
    以jQuery2.1.3的ajax方法为例： 

     $.ajax({
     url:"",
     dataType:"jsonp",
         data:{
             params:""
             }
     }).done(function(data){
         //dosomething..
 })

    仅仅是客户端使用jsonp请求数据是不行的，因为jsonp的请求是放在script标签中的，和普通请求不同的地方在于，它请求到的是一段js代码，如果服务端返回了json字符串，那么浏览器就会报错。所以jsonp返回数据需要服务端做一些处理。

  6. 服务端返回数据处理
    要获取请求到的数据，就需要服务端做一些判断，当参数中带有callback属性时，返回的type要为application/javascript，把数据作为callback的参数执行。示例：

 if (typeof callback === 'string' && callback.length !== 0) {
     this.charset = 'utf-8';
     this.set('X-Content-Type-Options', 'nosniff');

     body = body

     .replace(/\u2029/g, '\\u2029');

     // the /**/ is a specific security mitigation for "Rosetta Flash JSONP abuse"
     // the typeof check is just to reduce client error noise
     body = '/**/ typeof ' + callback + ' === \'function\' && ' + callback + '(' + body + ');';   
     }

服务端设置Access-Control-Allow-Origin

  1. 这种方式只要服务端把response的header头中设置Access-Control-Allow-Origin为制定可请求当前域名下数据的域名即可。一般情况下设为即可。这样客户端就不需要使用jsonp来获取数据。
    关于Access-Control-Allow-Origin是否安全，有待讨论。
  2. 浏览器支持：Access-Control-Allow-Origin是HTML5新增的一想标准，IE10以下是不支持的，所以如果产品面向的是PC端，就要使用服务端代理或者JSONP。
