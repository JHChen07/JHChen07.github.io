事件以及事件流

事件，就是文档或浏览器窗口中发生的一些交互瞬间（用户或浏览器本身执行的某种动作）。事件流描述的是从页面中接收事件的顺序。

  1. IE浏览器中规定的是事件冒泡流，事件冒泡即事件最开始由最具体的元素（文档中嵌套层次最深的节点）接收，然后逐级向上传播至最不具体的节点（文档）;

  2. Netscape浏览器中规定的是事件捕获流，事件捕获指不太具体的节点应该更早接收到事件，而最具体的节点最后接收到事件。事件捕获的用意在于事件到达预定节点之前捕获它。
    为了兼容更多的浏览器，非特殊情况一般我们都是把事件添加到在事件冒泡阶段。

    DOM事件流

    “DOM2级事件”规定的事件流包括3个阶段，事件捕获阶段、处于目标阶段和事件冒泡阶段。其中，事件捕获为事件截获提供机会；处于目标阶段时实际的目标接收事件；冒泡阶段能对事件作出响应。实际上，捕获阶段中事件并未到达目标，下一个阶段才能到达目标，并且，目标被当做是事件冒泡阶段的一部分。

  3. 基于DOM0的事件，对于同一个dom节点而言，只能注册一个，之后注册的同种事件会覆盖之前注册的。利用这个原理我们可以解除事件：btn5.onclick＝null；其中this就是绑定事件的那个元素。
  4. 以这种方式添加的事件处理程序会在事件流的冒泡阶段被处理。

    DOM2级事件处理程序

  5. DOM2支持同一dom元素注册多个同种事件，事件发生的顺序按照添加的顺序依次触发（IE是相反的）；DOM2事件通过addEventListener和removeEventListener管理。
  6. addEventListener(eventName,handlers,boolean)；removeEventListener()，两个方法都一样接收三个参数，第一个是要处理的事件名（事件句柄的引用e.g click），第二个是事件处理程序，第三个值为false时表示在事件冒泡阶段调用事件处理程序，一般建议在冒泡阶段使用，这样可以兼容更多的浏览器，特殊情况才在捕获阶段；

  7. 注意：通过addEventListener()添加的事件处理程序只能用removeEventListener()来移除，并且移除时传入的参数必须与添加时传入的参数（包括事件处理函数）一样。 

     var btn2 = document.getElementById('btn2');
 var handlers = function () {
     console.log(this.id);
 }; // 函数引用
 // 确保了传入的事件处理函数是同一个函数对象
 btn2.addEventListener('click',handlers,false);
 btn2.removeEventListener('click',handlers.false);

    IE事件处理程序

  8. IE用了attachEvent()，detachEvent()，接收两个参数，事件名称（和DOM2级稍有不同）和事件处理程序，通过attachEvent()添加的事件处理程序都会被添加到冒泡阶段，IE8及以前只支持事件冒泡。与addEventListener（）一样，attachEvent（）也可以添加多个事件处理程序。但是执行顺序却是与添加的顺序相反的顺序。

  9. IE事件处理与DOM0级方法的主要区别在于事件处理程序的作用域。后者是在其所属元素的作用域内运行，而前者则是会在全局作用域中进行，因此，此时的this为window。

    跨浏览器的事件处理程序

    事件对象

  10. 兼容触发DOM上的某个事件时，会产生一个事件对象event，这个对象中包含了所有与事件有关的信息，比如导致事件的元素target，事件的类型，及其他特定的相关信息。例如鼠标操作导致的事件对象中会包含鼠标的位置，单双击等，而键盘操作导致的事件对象会包含按下的键等信息；
  11. 事件被触发时，会默认给事件处理程序传入一个参数e , 表示事件对象；通过e，我们可以获得其中包含的与事件有关的信息；
  12. 只有在事件处理程序执行期间，event对象才会存在，一旦事件处理程序执行完毕，event对象就会被销毁；
  13. Event 对象代表事件的状态，比如事件在其中发生的元素、键盘按键的状态、鼠标的位置、鼠标按钮的状态。事件通常与函数结合使用，函数不会在事件发生前被执行！

    事件委托

  17. 每当将事件处理程序制定给元素时，运行中的浏览器代码与支持页面交互的JS代码之间就会建立一个连接，而这种连接越多，页面执行起来就越慢。考虑内存和性能问题，为了解决事件处理程序过多的问题，采用事件委托变得很有必要。（考虑到内存，也应该尽量减少不必要的事件处理程序，并且，对于内存中过时不用（删除了目标元素，但是相应的事件处理程序并没有移除）的“空事件处理程序”，也是很有必要将其移除的）
  18. 因为冒泡机制，比如既然点击子元素，也会触发父元素的点击事件，那我们完全可以将子元素的事件要做的事写到父元素的事件里，也就是将子元素的事件处理程序写到父元素的事件处理程序中，这就是事件委托；利用事件委托，只指定一个事件处理程序，就可以管理某一个类型的所有事件。

  19. 因为把事件绑定到了父节点上，因此省了绑定事件。就算后面新增的子节点也有了相关事件，删除部分子节点不用去销毁对应节点上绑定的事件。
    父节点是通过event.target来找对应的子节点的。（事件处理程序中的this值始终等于currentTarget的值，指向的是绑定到的那个元素）
    如： 

     var myLinks=document.getElementById('my-links');
 myHandlers=function(event){
     event=EventUtil.getEvent(event);
     var target=EventUtil.getTarget(event);

     switch(target.id){
         case 'item-1':
             location.href='http://xxxxxx/';
             break;
         case 'item-2':
             document.title='event';
             break;
         case 'item-3':
             console.log('hi');
             break;
     }
 };
 EventUtil.addHandler(myLinks,'click',myHandlers);

正则表达式是用于匹配字符串中字符组合的模式。在JS中，正则表达式也是对象。
正则表达式中，可以进行“分组”，用一对圆括号（捕获括号）括起来的表达式就是一个分组，分组之后可以将各组子表达式的匹配结果分别处理。
正则表达式的匹配模式支持以下3个标志：

  1. g: 表示全局（global）模式，模式将被应用于多有字符串，发现第一个匹配项后不会停止；
  2. i: 表示不区分大小写（case-insestive）模式；
  3. m: 表示多行（multiline）模式，即在到达一行末尾时还会继续查找下一行中是否存在与模式匹配的项。

正则表达式中的元字符包括：([{\^$|?*+.}])，因此，如果想要匹配这些字符串，pattern中的所有元字符都必须转义。

正则表达式的定义

字面量形式

var expression1 = /pattern/flags;

构造函数形式

RegExp的构造函数接收两个参数，一个是要匹配的字符串模式，另一个是可选的标志字符串。
var expression2 = new RegExp(‘[bc]at’, ‘i’);
以上，expression1和expression2是完全等价的正则表达式，但是传递给RegExp构造函数的两个参数都是字符串。（注意元字符的双重转义）
但是，字面量形式的正则表达式始终共享一个RegExp实例，实例属性不会重置，而构造函数创建的正则表达式，每一个新RegExp都是一个新实例，所以，第二种方法能循环调用实例属性，而第一种方法则不能。 

var re = null;
    i;
for(i=0; i<10; i++) {
    re = /cat/g;
    re.test('catastrongophe');
}
for(i=0;i<10;i++) {
    re = new RegExp('cat', 'g');
    re.test('catastrongophe');
}

RegExp实例属性

  1. global：布尔值，表示是否设置了g标志；
  2. ignoreCase：布尔值，表示是否设置了i标志；
  3. lastIndex：整数，表示开始搜索下一个匹配项的字符位置；
  4. multiline：布尔值，表示是否设置了m标志；
  5. source：正则表达式的字符串表示，按照字面量形式而传入构造函数中的字符串模式返回；

RegExp实例方法

exec()

exec()是专门为捕获组而设计的，接收一个参数，即要应用模式的字符串，返回包含第一的匹配项信息的数组，或者null；返回的数组虽然是Array的实例，但是有index和input属性，前者表示匹配项在字符串中的位置，后者表示应用模式的字符串。在返回的数组中，第一项是与整个模式匹配的字符串，其他项是与模式中捕获组匹配的字符串。若无捕获组则该数组只有第一项或为null。并且，该方法忽略g标志，但是每次匹配lastIndex属性会相应改变。

test()

test()接收字符串作为唯一的一个参数，返回true或false。适用于仅判断目标字符串与某个模式是否匹配。（验证输入等）

RegExp构造函数属性

长属性名 短属性名 说明
input $_ 最近一次要匹配的字符串
lastMatch $& 最近一次的匹配项
lastParen $+ 最近一次匹配的捕获组
leftContext $` input字符串中lastMatch之前的文本
multiline $* 布尔值，标志是否所有的表达式都使用多行模式
rightContext $’ input字符串中lastMatch之后的文本

注意：短属性名必须通过方括号访问：RegExp.[‘$&’]，另外还有9个用于存储不厚重的构造函数属性，RegExp.$1、…RegExp.$9，分别用于存储第1到第9个捕获组。

模式的局限性

下面列举了ECMAScript正则表达式不支持的特性：

  1. 匹配字符串开始和结尾的锚（但支持以插入^\&来匹配字符串的开始和结尾）；
  2. 向后查找（完全支持向前查找）；
  3. 并集和交集；
  4. 原子组；
  5. Unicode支持（除单个字符外，如\uffff；
  6. 命名的捕获组（但支持编号的捕获组）；
  7. s(single，单行)和x（free-spacing，无间隔）匹配模式；
  8. 条件匹配；
  9. 正则表达式注释。

快速理解语义化

语义化的含义就是用正确的标签做正确的事情，html语义化就是让页面的内容结构化，便于对浏览器、搜索引擎解析；在没有样式CCS情况下也以一种文档格式显示，并且是容易阅读的。搜索引擎的爬虫依赖于标记来确定上下文和各个关键字的权重，利于SEO（搜索引擎优化）。使阅读源代码的人对网站更容易将网站分块，便于阅读维护理解。
总结：

  1. 正确的标签做正确的事；
  2. 页面内容结构化；
  3. 无CSS也容易阅读，便于阅读维护和理解；

  4. 便于浏览器、搜索引擎解析。利于爬虫标记、利于SEO。

    HTML语义化标签

    对于HTML体系而言，web语义化是指使用语义恰当的标签，使页面具有良好的结构，让元素有含义。

DOCTYPE和浏览器渲染模式

一个文档类型标记是一种标准通用标记语言的文档类型声明，目的是告知通用标记语言解析器，应该是用什么文档类型定义（DTD）来解析文档。DOCTYPE还会对浏览器的渲染模式产生影响，不同的渲染模式会影响到浏览器对于CSS代码甚至是JavaScript脚本的解析，尤其是在IE系列浏览器中。因此，由DOCTYPE所决定的HTML页面的渲染模式至关重要。

浏览器解析HTML方式

  1. 非怪异（标准）模式 页面按照HTML和CSS的定义渲染
  2. 怪异模式 尝试模拟旧浏览器的行为
  3. 部分怪异（近乎标准）模式 除了实施一种表单元格尺寸的怪异行为，其他符合标准
    注：一个不含任何DOCTYPE的网页将会以怪异模式渲染。

    DOM结构

    DOM：Document Object Module，文档对象模型。

    W3C DOM标准

  4. 核心DOM - 针对任何结构化文档的标准模型
  5. XML DOM - 针对XML文档的标准模型
  6. HTML DOM - 针对HTML文档的标准模型

    DOM节点

    根据W3C的HTML DOM标准，HTML文档中的所有内容都是节点
    文档节点、元素节点、属性节点、文本节点、注释节点等。

    HTML DOM节点树

    HTML文本会被解析为DOM树，树中的所有节点均可通过JavaScript进行访问。所有HTML元素（节点）均可被修改，也可以创建或删除节点。

    节点的关系

    父、子、同胞等术语用于描述这些关系。节点树中，顶端节点被称为根；每个节点都有父节点，除了根；一个节点可以拥有任意数量的子，同胞拥有相同的根。

    盒模型

    每个元素都是一个矩形盒子，盒模型是元素大小的呈现方式。每个盒子有四个边：外边距边，边匡边，内填充边与内容边。

    盒子模型分为两种，一种是W3C标准盒模型，一种是IE盒模型。
  7. content-box （标准盒模型）
    width = content宽度
    height = content高度
  8. border-box （IE盒模型）
    width = border + padding + content宽度
    height = border + padding + content高度
    以上两者可以通过box-sizing属性自由转换，默认为content-box。

    JavaScript如何设置获取盒模型对应的宽和高

  9. dom.style.width/height 只能取到行内样式的宽和高，style标签中和link外链的样式取不到。
  10. dom.currentStyle.width/height 取到的是最终渲染后的宽和高，只有IE支持此属性。
  11. window.getComputedStyle(dom).width/height 同（2）但是多浏览器支持，IE9以上支持。
  12. dom.getBoundingClientRect().width/height 也是得到渲染后的宽和高，大多浏览器支持。IE9以上支持，除此外还可以取到相对于视窗的上下左右的距离

    外边距重叠

    当两个或更多个垂直边距相遇时，它们将形成一个外边距。这个外边距的高度等于两个发生叠加的外边距的高度中的较大者。但是只有普通文档流中的快款的垂直外边距才会发生外边距叠加。行内框、浮动框或绝对定位框之间的外边距不会叠加。
    垂直外边距叠加有三种情况：
  13. 元素自身叠加。当元素没有内容、内边距、边框时，它的上下边距就相遇了，即会在垂直方向产生叠加。当添加上述任何一项，叠加就会取消。
  14. 相邻元素叠加。
  15. 包含（父子）元素叠加。包含元素的外边距隔着父元素的内边距和边框，当这两项都不存在时，父子元素垂直外边距相邻，产生叠加。添加任何一项就会取消叠加。

    BFC

    BFC（Block Formatting Context）：块格式化上下文。
    BFC决定了元素如何对其内容进行定位，以及与其他元素的关系和相互作用。当设计到可视化布局时，BFC提供了一个环境，HTML元素在这个环境按照一定的规则进行布局。一个环境中的元素不会影响到其它环境中的布局。

    BFC原理（渲染规则）

  16. BFC元素垂直方向的边距会发生重叠。属于不同BFC外边距不会发生重叠。
  17. BFC的区域不会与浮动元素的布局重叠。
  18. BFC元素是一个独立的容器，外面的元素不会影响里面的元素。反之亦然。
  19. 计算BFC高度时，浮动元素也会参与计算（清除浮动）

    如何创建BFC

  20. 根元素或包含根元素的元素
  21. 浮动元素（元素的 float 不是 none）
  22. 绝对定位元素（元素的 position 为 absolute 或 fixed）
  23. 行内块元素（元素的 display 为 inline-block）
  24. 表格单元格（元素的 display 为 table-cell，HTML表格单元格默认为该值）
  25. 表格标题（元素的 display 为 table-caption，HTML表格标题默认为该值）
  26. 匿名表格单元格元素（元素的 display 为 table、table-row、 table-row-group、table-header-group、table-footer-group（分别是HTML table、row、tbody、thead、tfoot 的默认属性）或 inline-table）
  27. overflow 值不为 visible 的块元素
  28. display 值为 flow-root` 的元素
  29. contain 值为 layout、content 或 strict 的元素
  30. 弹性元素（display为 flex 或 inline-flex 元素的直接子元素）
  31. 网格元素（display 为 grid 或 inline-grid 元素的直接子元素）
  32. 多列容器（元素的 column-count 或 column-width 不为 auto，包括 column-count 为 1）
    column-span 为 all 的元素始终会创建一个新的BFC，即使该元素没有包裹在一个多列容器中（标准变更，Chrome bug）
    注：块格式化上下文包含创建它的元素内部的所有内容。浮动定位和清除浮动时只会应用于同一个BFC内的元素。浮动不会影响其他BFC中元素的布局，而清除浮动只能清除同一BFC中在它前面的元素的浮动。外边距折叠（Margin Collapsing）也只会发生在属于同一BFC的块级元素之间。

    CSS普通流（文档流）

    元素按照其在HTML中的位置顺序决定排布的过程。

    调节普通流元素位置

    一般使用margin是用来隔开元素与元素的间距；padding是用来隔开元素与内容的间隔。margin用于布局分开元素使元素与元素互不相干；padding用于元素与内容之间的间隔，让内容（文字）与（包裹）元素之间有一段“距离”。
    注：只要不是float和绝对定位方式布局的，都在普通流里面。

    变量对象

    JavaScript的基本类型

  33. Number
  34. String
  35. Boolean
  36. Undefined
  37. Null
  38. Object
    五种基本类型保存在内存的栈中，值固定，复制其变量时会创建这个值得一个副本。使用typeof区分。前五中没有原型没有构造函数。

    引用类型

    引用类型的值是对象，保存在堆中。引用类型的变量实际上是一个指针，他保存在栈中，指向堆内存中的对象。复制引用类型变量实际是复制该指针，所以他们都指向同一个对象，用instanceof确定哪一个值是哪种引用类型。

    内置对象、原生对象及宿主对象

    内置对象和元素对象是被ECMAScript规范定义和实现的，两者之间的差异微不足道。所有ECMAScript实现的对象都是原生对象（其中一些是内置对象、一些在程序执行的时候创建，例如用户自定义对象）。内置对象是原生对象的一个子集、是在程序开始之前内置到ECMAScript里的（例如，parseInt, Match等）。所有的宿主对象是由宿主环境提供的，通常是浏览器，并可能包括如window、alert等。
    注意，宿主对象可能是ES自身实现的，完全符合规范的语义。从这点来说，他们能称为“原生宿主”对象（尽快很理论），不过规范没有定义“原生宿主”对象的概念。

    特殊包装类

  39. Boolean: 布尔对象
  40. String: 字符串对象

  41. Number：数字对象
    这些对象的创建，是通过相应的内置构造器创建，并且包含原生值作为其内部属性，这些对象可以转换成原始值，反之亦然。 

     var c = new Boolean(true);
 var d = new String('test');
 var e = new Number(10);

 // 转换成原始值
 // 使用不带new关键字的函数

产生跨域的原因

跨域问题是浏览器同源策略限制，当前域名的js只能读取同域下的窗口属性

同源策略

定义

如果两个页面拥有相同的协议（protocol），端口和主机，那么这两个页面就属于同一个源（origin）
一个网站的网址组成包括协议名，子域名，主域名，端口号。比如https://github.com/，其中https是协议名，www是子域名，github是主域名，端口号是80，当在页面中从URL请求数据时，如果这个URL的协议名、子域名、主域名、端口号任意有一个不同，就会产生跨域问题。即使是在 http://localhost:80/ 页面请求 http://127.0.0.1:80/ 也会有跨域问题。

同源策略用意

同源策略限制了从同一个源加载的文档或脚本如何与来自另一个源的资源进行交互。这是一个用于隔离潜在恶意文件的重要安全机制。

解决跨域问题

解决跨域问题有以下几种方式：

  1. 使用JSONP
  2. 服务端代理
  3. 服务端设置Request Header头中Access-Control-Allow-Origin未指定可获取数据的域名

    JSONP的解决方式

  4. 原理
    JSONP解决跨域问题的原理是，浏览器的script标签是不受同源策略限制（可以在网页中设置script的src属性为cdn服务器中静态文件的路径），这样就可以使用script标签从服务器获取数据，请求时添加一个参数为callback=？，？号是要执行的回调方法。

  5. 前端实现
    以jQuery2.1.3的ajax方法为例： 

     $.ajax({
     url:"",
     dataType:"jsonp",
         data:{
             params:""
             }
     }).done(function(data){
         //dosomething..
 })

    仅仅是客户端使用jsonp请求数据是不行的，因为jsonp的请求是放在script标签中的，和普通请求不同的地方在于，它请求到的是一段js代码，如果服务端返回了json字符串，那么浏览器就会报错。所以jsonp返回数据需要服务端做一些处理。

  6. 服务端返回数据处理
    要获取请求到的数据，就需要服务端做一些判断，当参数中带有callback属性时，返回的type要为application/javascript，把数据作为callback的参数执行。示例：

    express4.12.3关于JSONP的实现代码： 

     // jsonp
 if (typeof callback === 'string' && callback.length !== 0) {
     this.charset = 'utf-8';
     this.set('X-Content-Type-Options', 'nosniff');
     this.set('Content-Type', 'text/javascript');

     // restrict callback charset
     callback = callback.replace(/[^\[\]\w$.]/g, '');

     // replace chars not allowed in JavaScript that are in JSON
     body = body
     .replace(/\u2028/g, '\\u2028')
     .replace(/\u2029/g, '\\u2029');

     // the /**/ is a specific security mitigation for "Rosetta Flash JSONP abuse"
     // the typeof check is just to reduce client error noise
     body = '/**/ typeof ' + callback + ' === \'function\' && ' + callback + '(' + body + ');';   
     }

服务端设置Access-Control-Allow-Origin

  1. 这种方式只要服务端把response的header头中设置Access-Control-Allow-Origin为制定可请求当前域名下数据的域名即可。一般情况下设为即可。这样客户端就不需要使用jsonp来获取数据。
    关于Access-Control-Allow-Origin是否安全，有待讨论。
  2. 浏览器支持：Access-Control-Allow-Origin是HTML5新增的一想标准，IE10以下是不支持的，所以如果产品面向的是PC端，就要使用服务端代理或者JSONP。

返回结果HTTP状态码

HTTP状态码负责表示客户端的HTTP请求的返回结果、标记服务器端的处理是否正常、通知出现的错误等工作。
数字中的第一位指定了相应类别，后两位无分类。响应的类别有以下5种：

代码 类别 原因短语
1xx informational (信息性状态码) 接收的请求正在处理
2xx success (成功状态码) 请求正常处理完毕
3xx redirection (重定向状态码) 需要进行附加操作以完成请求
4xx client error (客户端错误状态码) 服务器无法处理请求
5xx server error (服务器错误状态码) 服务器处理请求出错

只要遵守状态码类别的定义，即使改变RFC2616中定义的状态码，或服务器端自行创建状态码都没问题。

2xx 成功

200 OK

表示从客户端发来的请求在服务器端被正常处理了。但是在响应报文内，随状态码一起返回的信息会因为方法的不同而发生改变。

204 No Content

该状态码代表服务器接收的请求已经成功处理，但在返回的响应中不包含实体的主体部分。另外，也不允许返回任何实体的主体。一般在只需要向服务器发送信息，而不需要返回信息给客户端时使用。

206 Partial Content

该状态码表示客户端进行了范围请求，而服务器成功执行了这部分的GET请求。响应报文中包含由Content-Range指定范围的实体内容。

3xx 重定向

3xx响应结果表明浏览器需要执行某些特殊的处理以正确处理请求。

301 Moved Permanently

永久性重定向。该状态码表示请求的资源被分配了新的URI，同时，书签也应按需进行更新。

302 Found

临时性重定向。该状态码也表示请求的资源被分配了新的URI，但是临时性质的。并不会更新书签。

303 See Other

该状态码表示请求的资源存在另一个URI，应使用GET方法定向获取请求的资源。与302的区别是，303明确表示客户端应该用GET方法获取资源。
当返回301、302、303时，几乎所有的浏览器都会把POST改成GET，并删除请求报文的主体，之后请求会自动再次发送。但301、302标准时禁止把POST方法改成GET的。

304 Not Modified

该状态码表示客户端发送附带条件的请求时，服务器允许访问资源，但因发生请求未满足的条件的情况后，直接返回304 Not Modified（服务器资源未改变，可直接使用客户端未过期的资源），返回时，也不包含任何响应的主体部分。并且304和重定向没有什么关系。

307 Temperorary Redirect

临时重定向。与302有同样的含义，但307会按照标准，不会将POST改成GET。

4xx 客户端错误

400 Bad Request

表示请求报文中存在语法错误，需修改请求的内容再次发送请求。另外，浏览器会像200 OK一样对待该状态码。

401 Unauthrized

表示发送的请求需要有通过HTTP认证（BASIC 认证、DIGEST 认证的认证信息。若之前已进行过1次请求，则表示用户认证失败。
返回包含401的响应必须包含一个适用于被请求资源的WWW-Authenticate 首部以质询用户信息。当浏览器初次接收401响应，会弹出认证用的窗口。

403 Forbidden

表明请求的资源被服务器拒绝了。并且服务器没有必要给出详细的拒绝理由。但是可以在实体的主体部分对原因进行描述。未获得文件系统的访问授权，访问权限出现某些问题（从未授权的发送元IP地址试图访问）等都可能是403的原因。

403 Not Found

表明服务器端上没有请求的资源。除此之外，还可以在服务器拒绝请求且不想说明理由时使用。

5xx 服务器错误

501 Internal Server Error

表明服务器端在执行请求时发生了错误。

503 Service Unavailable

表明服务器端暂时处于超负载或正在停机维护，现在无法处理请求。
注：状态码有可能和实际状况不一致。
